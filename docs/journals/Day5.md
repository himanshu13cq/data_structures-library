# Daily Journal

**Date: 13/07/2026**

The first change I made was adding a third template parameter to the `HashMap`, allowing users to provide their own hash function. Instead of hardcoding the hashing logic inside `HashMap`, I moved it into a separate `DefaultHash<Key>` functor. This made the HashMap cleaner because its only responsibility is managing key-value pairs, while the hashing logic is handled independently. After that I tested the design with a custom `Employee` class and its own hashing functor, which confirmed that insertion, updating existing keys, collision handling through separate chaining, and rehashing all continued to work correctly.

While making this change I ran into an easy-to-miss template bug. I updated the class to use a third template parameter, but forgot to update a few member function definitions. Even though the function bodies were using the new template parameter, the `template<>` declarations still contained only two template parameters, so the compiler didn't know what the third parameter was. It was a good reminder that whenever a template class changes, every out-of-line function definition has to be updated as well.

The next thing I worked on was improving the default hash function. I added support for `double` values and learned why simply casting a floating-point number to an integer makes a poor hash function. Different floating-point values can become the same integer after truncation, creating unnecessary collisions. Instead, I hashed the actual bit representation of the `double`, which preserves much more information.

I also spent some time thinking about how a generic HashMap should behave for completely unknown key types. Ideally every custom type should have its own hash function, but I wanted the container to remain usable even if the user didn't provide one. I eventually decided to use the object's memory address as a fallback hash. Although this only gives identity-based behavior rather than value-based behavior, it at least allows unsupported types to be stored. To make incorrect usage easier to detect, I also added a compile-time check requiring the key type to implement `operator==`. Since I used a `requires` expression for this check, I upgraded the project from C++17 to C++20.

The biggest discovery today actually came from another data structure. While refactoring the HashMap to reuse my `DynamicArray` as bucket storage, I found a bug in `DynamicArray::append()`. The function was assigning into memory that had been allocated but never constructed. This never caused problems when storing primitive types like `int`, but as soon as I started storing `LinkedList` objects the bug became obvious because their assignment operator depends on properly initialized internal state.

Fixing that bug meant changing both `append()` and the resizing logic to construct objects directly using placement `new` instead of assigning into raw memory. Once that was corrected, I refactored the HashMap to store its buckets inside `DynamicArray`, removing a large amount of manual memory management code from the HashMap itself. The constructor, destructor, copy constructor, assignment operator, and `rehash()` all became much simpler because `DynamicArray` now manages the lifetime of the bucket array.

During the refactor I introduced another bug by manually destroying every bucket inside `rehash()` before assigning the new bucket array. I later realized that `DynamicArray`'s assignment operator already destroys its existing elements, so every bucket destructor was being called twice. Although the program appeared to work because `LinkedList` reset its internal pointers after destruction, I recognized that relying on this behavior is still undefined behavior. Removing the manual destructor calls fixed the problem and made ownership much clearer.

I also ran into another template syntax mistake while nesting templates. I forgot a closing `>` in a declaration like `DA<LinkedList<KV<Key, Value>>>`, and the compiler produced a long list of confusing errors that had nothing to do with the real problem. It reminded me that with deeply nested templates, it's worth slowing down and checking the matching angle brackets before assuming there's a more complicated issue.

Overall, today was less about implementing new algorithms and more about improving the overall design of the library. I made the HashMap more reusable through custom hash functions, found and fixed a hidden lifetime bug in `DynamicArray`, and reduced the amount of manual memory management by reusing components I had already built. More than anything, today reinforced how closely templates, object lifetime, and ownership are connected when building generic data structures from scratch.
